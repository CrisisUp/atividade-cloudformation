Relatório de Execução

Esta página documenta cada comando digitado no terminal Linux durante o laboratório. O objetivo é explicar a intenção técnica por trás de cada interação e o resultado observado no terminal.

20:14 Instalação e Verificação do Binário
[ec2-user@ip-172-31-25-103 ~]$ sudo yum install -y yum-utils
[ec2-user@ip-172-31-25-103 ~]$ sudo yum-config-manager --add-repo https://rpm.releases.hashicorp.com/AmazonLinux/hashicorp.repo
[ec2-user@ip-172-31-25-103 ~]$ sudo yum install -y terraform
[ec2-user@ip-172-31-25-103 ~]$ terraform version
Terraform v1.15.3
on linux_amd64

O que aconteceu:

Iniciamos com a configuração do repositório oficial da HashiCorp. O comando yum-config-manager é essencial para que o sistema saiba onde buscar o pacote oficial do Terraform. A verificação da versão confirma que estamos utilizando a v1.15.3, garantindo que as funcionalidades mais recentes de HCL estejam disponíveis.

20:16 Criação do Espaço de Trabalho
[ec2-user@ip-172-31-25-103 ~]$ mkdir -p ~/techstock-tf-lab && cd ~/techstock-tf-lab
[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ ls -la

O que aconteceu:

Isolamos o projeto em um diretório dedicado. O uso da flag -p no mkdir garante que o comando não falhe caso o diretório já exista. O ls -la inicial serve para confirmar que o diretório está vazio e pronto para receber os arquivos .tf.

20:24 - 20:27 Escrita dos Arquivos de Configuração (HCL)
[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ cat > provider.tf << 'EOF' ...
[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ cat > main.tf << 'EOF' ...
[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ cat > variables.tf << 'EOF' ...
[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ cat > outputs.tf << 'EOF' ...

O que aconteceu:

Utilizamos o comando cat com o redirecionador << 'EOF' (Here Document) para criar os arquivos sem precisar de um editor interativo. Isso é ideal para scripts de automação.

• provider.tf: Define o provedor AWS na região us-east-1.
• main.tf: Define a infraestrutura (AMI, Security Group e EC2).
• variables.tf: Parametriza o ambiente (instance_type, environment).
• outputs.tf: Expõe IDs e IPs após a criação.

20:29 - 20:31 Inicialização e Validação
[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ terraform init
Initializing the backend...
Initializing provider plugins...
Terraform has been successfully initialized!

[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ terraform validate
Success! The configuration is valid.

O que aconteceu:

O terraform init é o comando que prepara o diretório baixando o provider AWS (plugin). Sem ele, nenhum outro comando funciona. O terraform validate garante que não cometemos erros de sintaxe nos blocos HCL antes de tentar qualquer ação na nuvem.

20:31 - 20:35 Planejamento e Execução do Deploy
[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ terraform plan
Plan: 2 to add, 0 to change, 0 to destroy.

[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ terraform apply
Do you want to perform these actions?
Enter a value: yes
Apply complete! Resources: 2 added, 0 changed, 0 destroyed.

O que aconteceu:

O plan gerou a lista de ações: criar o Security Group e a Instância EC2. No apply, confirmamos com yes e o Terraform levou cerca de 22 segundos para provisionar os recursos reais na AWS. Os outputs exibidos no final (instance_id e sg_id) são os dados reais retornados pela API da AWS.

20:44 - 20:50 Atualização e Gerenciamento de Drift
[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ nano main.tf
[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ terraform plan
~ resource "aws_instance" "lab" {
    ~ tags = { + "Owner" = "aluno-lab" }
}

O que aconteceu:

Fizemos uma alteração "In-Place" (no mesmo lugar). Adicionamos uma tag de proprietário. O Terraform identificou que não precisa destruir a instância para aplicar essa mudança, apenas atualizar os metadados. Vimos o símbolo ~ indicando modificação.

20:52 - 20:54 Inspeção do Estado (State)
[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ terraform state list
aws_instance.lab
aws_security_group.lab

[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ terraform state show aws_instance.lab

O que aconteceu:

Navegamos pelo cérebro do Terraform: o terraform.tfstate. O comando state list confirma quais recursos o código está rastreando. O state show exibe detalhes técnicos que não escrevemos no código, mas que a AWS atribuiu, como o Private IP e o ARN.

20:56 - 20:59 Manipulação Avançada: State RM e Import
[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ terraform state rm aws_security_group.lab
[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ terraform plan
(Indica que criará um novo SG)

[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ terraform import aws_security_group.lab sg-0147decfcf43db838

O que aconteceu:

Realizamos uma "cirurgia" no estado. Primeiro, removemos o SG do controle do Terraform com state rm. Quando rodamos o plan, o Terraform pensou que o SG não existia e quis criar outro. Para resolver, usamos o import para re-adotar o recurso existente sem duplicá-lo.

21:01 - 21:04 Variáveis, Locals e Console Interativo
[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ terraform plan -var-file="dev.tfvars"
[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ terraform console
> local.full_name
"techstock-lab"

O que aconteceu:

Testamos a flexibilidade do código. Usamos -var-file para mudar o ambiente sem tocar no código principal. O terraform console foi usado para testar expressões HCL em tempo real, validando o bloco locals que calcula nomes de recursos dinamicamente.

21:05 - 21:11 Loops com for_each e Automação
[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ terraform apply -var-file="lab.tfvars" -auto-approve
aws_security_group.app_sgs["backend"]: Creating...
aws_security_group.app_sgs["monitoring"]: Creating...
aws_security_group.app_sgs["frontend"]: Creating...

O que aconteceu:

Escalamos a infraestrutura. Com um único bloco for_each, criamos três Security Groups simultaneamente. A flag -auto-approve foi usada para ignorar a pergunta "yes", tornando o deploy 100% automatizado.

21:13 - 21:17 Cleanup: O Ciclo de Destruição
[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ terraform plan -destroy -var-file="lab.tfvars"
Plan: 0 to add, 0 to change, 5 to destroy.

[ec2-user@ip-172-31-25-103 techstock-tf-lab]$ terraform destroy -var-file="lab.tfvars"
Destroy complete! Resources: 5 destroyed.

O que aconteceu:

Encerramos o laboratório. O plan -destroy serve como uma verificação de segurança para garantir que não vamos apagar nada acidentalmente. O destroy final removeu todos os 5 recursos (1 EC2 + 4 SGs), deixando a conta AWS limpa e sem custos pendentes.
