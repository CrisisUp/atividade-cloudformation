Visão Geral da Atividade

Este relatório documenta a execução da atividade prática de Terraform no ambiente AWS Learner Lab. Foram percorridas etapas de configuração inicial, workflow básico (Init, Plan, Apply), evolução de infraestrutura, gestão de estado, uso de variáveis avançadas, loops para escalabilidade e o processo final de destruição de recursos.

1
Configuração do Ambiente
Setup

Documentação — Instalação

1. Instalação de utilitários: O comando sudo yum install -y yum-utils instala um conjunto de ferramentas que estendem as funcionalidades do gerenciador de pacotes padrão (YUM).
2. Configuração do Repositório: Com sudo yum-config-manager --add-repo ..., incluímos a fonte de software oficial da HashiCorp na lista de confiança do Linux.
3. Instalação do Terraform: O comando sudo yum install -y terraform realiza o download e a instalação física do binário.
4. Verificação: terraform version confirma que o binário está funcional. Versão observada: v1.15.3.

Documentação — Workspace

1. Criação de Diretório: Uso de mkdir -p ~/techstock-tf-lab && cd ~/techstock-tf-lab para organizar os arquivos do projeto.
2. Listagem de Arquivos: O diretório contém arquivos fundamentais: provider.tf, main.tf, variables.tf e outputs.tf.
3. Pasta .terraform: Criada durante a inicialização para armazenar plugins dos provedores.
4. Arquivo de Estado: Presença do terraform.tfstate, o banco de dados que rastreia a infraestrutura.

Documentação — Provedor

1. Técnica Heredoc: Uso de cat > provider.tf << 'EOF' para escrever o arquivo de configuração via terminal.
2. Requisitos de Versão: Definição de required_version = ">= 1.6.0" para garantir compatibilidade.
3. Bloco AWS: Configuração da região us-east-1 e versão do provider ~> 5.0.
4. Segurança: As credenciais não estão no código, utilizando o perfil LabRole injetado pelo ambiente.

Documentação — Recursos e Inicialização

1. Data Sources: Localização dinâmica da AMI Amazon Linux 2023 mais recente.
2. Security Group: Definição das regras de firewall para a porta 80.
3. Dependências: Referência dinâmica vpc_security_group_ids = [aws_security_group.lab.id].
4. Init: terraform init prepara o diretório e baixa os plugins necessários.

Ciclo de Vida

Documentação — Validação e Plano Inicial

1. Validação Sintática: terraform validate confirma que o código HCL está estruturalmente correto.
2. Formatação: terraform fmt padroniza a indentação dos arquivos.
3. Detecção de Desvios: No plano, o Terraform detectou que uma instância foi removida manualmente da AWS e planejou sua recriação.
4. Símbolo (+): Indica que novos recursos serão criados para atingir o estado desejado.

Documentação — Execução do Apply

1. Provisionamento: Envio das ordens para a API da AWS para criar SG e EC2.
2. Orquestração: Criação automática do Security Group antes da instância devido à dependência de rede.
3. Outputs: Impressão imediata do ID da instância e do SG após a conclusão.

Resultados e Verificação

1. User Data: Automatização da instalação do Nginx através do script de inicialização.
2. Conectividade: Sucesso no acesso à página "Welcome to nginx!", validando as regras do SG.
3. Gestão de Outputs: Uso do comando terraform output para consultas rápidas de metadados.

Evolução de Infraestrutura
Mudanças

Documentação — Update In-Place vs Replace

1. Update In-Place (~): Adição de tags permitiu a atualização sem downtime do recurso.
2. Replace (-/+): A alteração do nome do Security Group forçou a destruição e recriação do mesmo.
3. Efeito Cascata: Como a EC2 dependia do SG, ela também foi marcada para atualização/substituição.
4. Gestão de Risco: O plano alertou sobre a destruição iminente antes da execução do apply.

Aplicando Evoluções

1. Execução: O apply atualizou o estado com a nova tag Owner: aluno-lab.
2. Validação: Verificação dos metadados via CLI para confirmar a aplicação da mudança.
3. Integridade: O arquivo .tfstate foi sincronizado com as novas definições de tags.
Checklist de Evolução
Identificado símbolo de update in-place (~)
Identificado símbolo de replacement (-/+)
Tags aplicadas e verificadas

Gestão de Estado (State)
Terraform State

Inspeção e Anatomia do State

1. State Show: Tradução do JSON complexo para o formato legível HCL com todos os metadados da AWS.
2. Estrutura JSON: Visualização bruta do arquivo contendo serial, lineage e detalhes técnicos.
3. Controle de Versão: O campo 'serial' atua como controle de concorrência e integridade.
4. Mapeamento: Vinculação entre o nome lógico no código e o ID físico real na nuvem.

Manipulação de Estado e Importação

1. State RM: Remoção de um recurso do estado sem deletá-lo fisicamente da AWS.
2. Desincronização: O plano detectou que o recurso "sumiu" e tentou criá-lo novamente.
3. Importação: Uso de terraform import para re-adotar o recurso existente de volta ao controle.
4. Sincronização Final: O plano retornou a "No changes" após a importação bem-sucedida.
Checklist de State

Variáveis e Dados Sensíveis

1. .tfvars: Separação entre lógica de infraestrutura e dados de configuração para diferentes ambientes.
2. Sensitive: Marcação de variáveis como sensíveis para mascarar valores em logs e no terminal.
3. Variáveis de Ambiente: Uso do prefixo TF_VAR_ para injeção segura de segredos.

Locals e Console Interativo

1. Locals: Implementação de constantes e lógicas de concatenação para manter o código DRY.
2. Console: Uso da ferramenta interativa para debugar funções e interpolações em tempo real.
3. Previsibilidade: O plano final validou a mudança do tipo de instância de t2.micro para t2.small.

3. Identificadores: Uso de chaves significativas (backend, frontend) em vez de índices numéricos.

Provisionamento em Massa
