Primeiro Resource — EC2 com HCL
HCL Basico ~20 min
ℹ️ Voce vai escrever seu primeiro arquivo HCL, executar o ciclo completo init → validate → plan → apply e entender o que o Terraform faz em cada etapa.
HCL vs YAML (CloudFormation)
CloudFormation usa YAML/JSON com funcoes como !Ref e !Sub.
Terraform usa HCL (HashiCorp Configuration Language) com referencias diretas: var.nome, resource.tipo.nome.atributo.
HCL suporta logica real: loops (for_each), condicionais (count), funcoes embutidas.

Passo 1 — Escreva o provider.tf

No CloudShell, use o editor nano para criar os arquivos:

CloudShell
cd ~/techstock-tf-lab

# Cria o provider.tf
cat > provider.tf << 'EOF'
terraform {
  required_version = ">= 1.6.0"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = "us-east-1"
  # Credenciais vem automaticamente do CloudShell (LabRole)
  # NUNCA coloque access_key e secret_key no codigo!
}
EOF

cat provider.tf

Passo 2 — Escreva o main.tf com um Security Group e EC2

CloudShell
cat > main.tf << 'EOF'
# Data source: busca a AMI mais recente do Amazon Linux 2023
# "data" nao cria recurso -- apenas busca informacao existente na AWS
data "aws_ami" "al2023" {
  most_recent = true
  owners      = ["amazon"]
  filter {
    name   = "name"
    values = ["al2023-ami-2023*-x86_64"]
  }
}

# Security Group: aceita porta 80 de qualquer lugar
resource "aws_security_group" "lab" {
  name        = "techstock-tf-lab-sg"
  description = "SG para o lab de Terraform TechStock"

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
    description = "HTTP publico"
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"    # -1 = todos os protocolos
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Name      = "techstock-tf-lab-sg"
    ManagedBy = "Terraform"
  }
}

# EC2: usa a AMI do data source e o SG criado acima
resource "aws_instance" "lab" {
  # Referencia ao data source -- nao e hardcode!
  ami           = data.aws_ami.al2023.id
  instance_type = var.instance_type  # vem de variables.tf

  # Referencia ao recurso acima -- cria dependencia automatica
  # Terraform garante que o SG e criado ANTES da EC2
  vpc_security_group_ids = [aws_security_group.lab.id]

  iam_instance_profile = "LabInstanceProfile"

  tags = {
    Name        = "techstock-tf-lab-ec2"
    ManagedBy   = "Terraform"
    Environment = var.environment
  }
}
EOF

cat main.tf

EOF

O Plan como Ferramenta de Revisao — Diff antes de Aplicar
terraform plan ~15 min
ℹ️ O terraform plan e o diferencial do Terraform em relacao ao CloudFormation. Ele mostra exatamente o que vai mudar antes de qualquer acao. Nesta tarefa voce vai fazer mudancas e interpretar o plano antes de aplicar.
Interpretar o terraform plan
+ create — recurso sera criado (nao existia)
~ update in-place — recurso sera modificado sem recriar (sem downtime)
- destroy — recurso sera deletado
-/+ replace — recurso sera DELETADO e RECRIADO (possivel downtime — cuidado!)

Passo 1 — Mudanca de tag (update in-place)

Adicione uma tag nova na EC2 e veja que o plan mostra update sem recriar:

CloudShell — edite o main.tf
cd ~/techstock-tf-lab

# Edite o main.tf com nano
nano main.tf

# No bloco tags do aws_instance.lab, adicione:
#   Owner = "aluno-lab"
# O bloco tags deve ficar assim:
#   tags = {
#     Name        = "techstock-tf-lab-ec2"
#     ManagedBy   = "Terraform"
#     Environment = var.environment
#     Owner       = "aluno-lab"     # LINHA NOVA
#   }
# Salve: Ctrl+X → Y → Enter

# Agora execute o plan e leia a saida
terraform plan
# aws_instance.lab will be updated in-place
~ resource "aws_instance" "lab" {
~ tags = {
+ "Owner" = "aluno-lab"
# (3 unchanged attributes hidden)
}
# (15 unchanged attributes hidden)
}
Plan: 0 to add, 1 to change, 0 to destroy.

O simbolo ~ significa update in-place. A EC2 nao sera recriada — apenas a tag sera adicionada. Execute terraform apply para aplicar.

Passo 2 — Mudanca que causa replace (cuidado!)

Mude o nome do Security Group e veja o que acontece no plan:

CloudShell
nano main.tf

# Mude o name do aws_security_group.lab:
#   name = "techstock-tf-lab-sg-v2"    # era "techstock-tf-lab-sg"
# Salve: Ctrl+X → Y → Enter

terraform plan
# aws_security_group.lab must be replaced
-/+ resource "aws_security_group" "lab" {
~ name = "techstock-tf-lab-sg" → "techstock-tf-lab-sg-v2" # forces replacement
}
# aws_instance.lab must be replaced
Plan: 2 to add, 0 to change, 2 to destroy.
⚠️ forces replacement significa que o SG sera DELETADO e RECRIADO. Como a EC2 depende do SG, ela tambem sera recriada. Em producao isso causaria downtime! Reverta a mudanca no nome antes de continuar.
CloudShell — reverta o nome do SG
nano main.tf
# Volte o name para "techstock-tf-lab-sg"
# Ctrl+X → Y → Enter

terraform plan
# Agora deve mostrar: No changes. Infrastructure is up-to-date.
Checklist
Plan de tag nova mostrou ~ 1 to change sem destroy
Plan de nome do SG mostrou -/+ replace e forces replacement
Entende a diferenca entre update in-place e replace
Nome do SG revertido: plan mostra No changes

State — O Que o Terraform Rastreia
terraform state ~15 min
ℹ️ O state file e o mapa entre seu codigo HCL e os recursos reais na AWS. Sem ele, o Terraform nao saberia o que ja criou. Nesta tarefa voce vai inspecionar, importar e manipular o state.
Por que o state existe?
Diferente do CloudFormation (que guarda o estado na AWS), o Terraform guarda localmente em terraform.tfstate.
O state mapeia: "o recurso aws_instance.lab no meu codigo corresponde ao recurso i-0abc123456 na AWS".
Sem o state: terraform apply criaria tudo do zero toda vez, causando duplicatas.

Passo 1 — Inspecione o state

CloudShell
cd ~/techstock-tf-lab

# Lista todos os recursos no state
terraform state list

# Detalha um recurso especifico (substitua pelo nome que apareceu)
terraform state show aws_instance.lab

# O output mostra TODOS os atributos: id, ami, instance_type,
# public_ip, private_ip, tags, vpc_security_group_ids, etc.

# Ve o arquivo de state diretamente (JSON legivel)
cat terraform.tfstate | python3 -m json.tool | head -60

Passo 2 — Simule drift e veja o plan detectar

  1. EC2 → Instances → techstock-tf-lab-ec2 → Actions → Instance settings → Edit tags · Adicione tag: Key DriftTag Value adicionada-fora-do-tf → Save
  2. De volta ao CloudShell: terraform plan
  3. O plan deve mostrar que a tag DriftTag sera removida ao aplicar (o TF quer que o estado real fique igual ao codigo)
  4. Execute terraform apply -auto-approve · O Terraform remove a tag que voce adicionou manualmente
✅ O Terraform detecta que a realidade divergiu do codigo e corrige na proxima aplicacao. Isso e o comportamento esperado — o codigo e a fonte da verdade.

Passo 3 — Remover do state sem deletar o recurso

CloudShell
# Remove o SG do state (o SG continua existindo na AWS!)
terraform state rm aws_security_group.lab

# Verifica: o SG sumiu do state
terraform state list

# Agora execute o plan: Terraform quer CRIAR o SG de novo
# (ele nao sabe que o SG ja existe)
terraform plan

# Para re-adicionar ao state sem recriar, use o import
# O argumento e o ID fisico do recurso na AWS
SG_ID=$(aws ec2 describe-security-groups \
  --filters Name=group-name,Values=techstock-tf-lab-sg \
  --query 'SecurityGroups[0].GroupId' \
  --output text)
echo "SG ID: $SG_ID"

terraform import aws_security_group.lab $SG_ID

# Agora o plan deve voltar a mostrar No changes
terraform plan
Checklist
terraform state list mostra os 3 recursos gerenciados
Tag adicionada manualmente foi detectada no plan como mudanca a remover
terraform state rm removeu o SG do state sem deletar da AWS
terraform import re-adicionou o SG ao state
Plan voltou a mostrar No changes

for_each — Criando Multiplos Recursos com Loop
HCL Avancado ~15 min
ℹ️ Uma das maiores vantagens do HCL sobre o YAML do CloudFormation: loops nativos com for_each e count. Em vez de copiar e colar 5 blocos de Security Group, voce usa um loop.
for_each vs CloudFormation
No CloudFormation: para criar 3 SGs voce copia o bloco 3 vezes (ou usa Macros avancadas).
No Terraform: um bloco com for_each cria quantos recursos o mapa tiver.

Crie multiplos SGs com for_each

CloudShell — crie um novo arquivo para o exercicio
cat > sgs.tf << 'EOF'
# Map de Security Groups: nome => porta
variable "app_ports" {
  type = map(number)
  default = {
    backend    = 3000
    monitoring = 9090
    frontend   = 80
  }
}

# Um bloco cria TRES Security Groups
resource "aws_security_group" "app_sgs" {
  # for_each itera sobre cada entrada do map
  for_each = var.app_ports

  name        = "techstock-${each.key}-sg"
  description = "SG para ${each.key} porta ${each.value}"

  ingress {
    from_port   = each.value  # each.value = a porta do map
    to_port     = each.value
    protocol    = "tcp"
    cidr_blocks = ["10.0.0.0/16"]
    description = "${each.key} interno"
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Name      = "techstock-${each.key}-sg"
    ManagedBy = "Terraform"
    Port      = each.value  # each.key = nome (backend/monitoring/frontend)
  }
}

# Output: mapa de nome → ID do SG
output "sgs_criados" {
  description = "IDs dos Security Groups criados pelo for_each"
  value = {
    for name, sg in aws_security_group.app_sgs :
    name => sg.id
  }
}
EOF

terraform plan -var-file="lab.tfvars"
# Plan deve mostrar 3 SGs para criar (+ o sg.lab ja existente = no change)

terraform apply -var-file="lab.tfvars" -auto-approve
✅ Esperado: Resources: 3 added. Output sgs_criados mostra um mapa com os 3 IDs. Na Console, 3 novos SGs: techstock-backend-sg, techstock-monitoring-sg, techstock-frontend-sg.
