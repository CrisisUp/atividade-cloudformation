1
Primeira Stack — EC2 pelo Console
Console ~15 min
ℹ️ Voce vai criar uma stack CloudFormation diretamente pelo Console AWS. Sem CLI, sem arquivo local — o template YAML e colado direto no editor do Console.
💡 O que e uma Stack?
Uma stack e um conjunto de recursos AWS gerenciados como uma unidade pelo CloudFormation. Ao criar a stack, o CF cria todos os recursos na ordem certa. Ao deletar a stack, ele apaga tudo junto.

Passo 1 — Acesse o CloudFormation e inicie a criacao

  1. AWS Console → CloudFormation → Create stack → With new resources (standard)
  2. Prerequisite: Template is ready
  3. Template source: Enter template in Designer → clique em Create template in Designer
  4. No Designer, clique no botao Template (aba inferior) → apague o conteudo e cole o YAML abaixo
AWSTemplateFormatVersion: '2010-09-09'
Description: TechStock Lab - EC2 simples via Console

Parameters:
  InstanceType:
    Type: String
    Default: t2.micro
    AllowedValues: [t2.micro, t2.small]
    Description: Tipo da instancia EC2

Resources:
  LabEC2:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: ami-0c02fb55956c7d316
      InstanceType: !Ref InstanceType
      IamInstanceProfile: LabInstanceProfile
      Tags:
        - Key: Name
          Value: !Sub "${AWS::StackName}-ec2"
        - Key: ManagedBy
          Value: CloudFormation

Outputs:
  InstanceId:
    Value: !Ref LabEC2
    Description: ID da instancia criada
  PrivateIP:
    Value: !GetAtt LabEC2.PrivateIp
    Description: IP privado da instancia
📸 O Designer deve mostrar o recurso LabEC2 (icone de EC2) no canvas. Se aparecer erro de sintaxe, revise a indentacao do YAML.

Passo 2 — Revisar e criar

  1. No Designer, clique no botao ☁️ Create stack (icone de nuvem com seta) no topo → Next
  2. Stack name: techstock-lab-cf → Next
  3. Parameters: InstanceType → mantenha t2.micro → Next
  4. Configure stack options: nao altere nada → Next
  5. Review: role ate o final → Create stack
  6. CloudFormation → Stacks → techstock-lab-cf → Events · Acompanhe os eventos em tempo real. Aguarde CREATE_COMPLETE
✅ Esperado: Status da stack: CREATE_COMPLETE. Aba Outputs mostra InstanceId e PrivateIP preenchidos automaticamente.
Checklist
Stack criada com status CREATE_COMPLETE
Aba Events mostra a criacao do recurso LabEC2
Aba Outputs exibe InstanceId (i-xxxx) e PrivateIP
EC2 visivel em EC2 → Instances com tag ManagedBy: CloudFormation

Parameters e Mappings — Infra Parametrizada
Parameters ~20 min
ℹ️ Agora voce vai criar um template mais completo com Parameters (entradas dinamicas) e Mappings (dicionario de valores por ambiente). Isso permite reutilizar o mesmo template para dev, staging e prod mudando apenas os parametros.
💡 Por que Parameters e Mappings?
Parameters sao entradas que o usuario fornece ao criar a stack (ex: senha do banco, tipo de instancia). Mappings sao dicionarios fixos no template — usados para variar valores por regiao ou ambiente sem precisar de parametros.

Passo 1 — Crie a nova stack com template parametrizado

  1. CloudFormation → Stacks → Create stack → With new resources
  2. Template source: Enter template in Designer → clique em Create template in Designer
  3. Cole o template abaixo na aba Template do Designer:
AWSTemplateFormatVersion: '2010-09-09'
Description: TechStock - Template com Parameters e Mappings

Parameters:
  Environment:
    Type: String
    Default: lab
    AllowedValues: [lab, dev, prod]
    Description: Ambiente de implantacao

  AppName:
    Type: String
    Default: techstock
    Description: Nome base da aplicacao

Mappings:
  InstanceSize:
    lab:  { EC2: t2.micro  }
    dev:  { EC2: t2.small  }
    prod: { EC2: t2.medium }

  AMIByRegion:
    us-east-1: { AL2023: ami-0c02fb55956c7d316 }

Conditions:
  IsProd: !Equals [!Ref Environment, prod]

Resources:
  AppInstance:
    Type: AWS::EC2::Instance
    Properties:
      # !FindInMap busca o tipo correto para o ambiente escolhido
      InstanceType: !FindInMap [InstanceSize, !Ref Environment, EC2]
      # !FindInMap busca a AMI correta para a regiao atual
      ImageId: !FindInMap [AMIByRegion, !Ref AWS::Region, AL2023]
      IamInstanceProfile: LabInstanceProfile
      Tags:
        - Key: Name
          # !Sub substitui variaveis na string
          Value: !Sub "${AppName}-${Environment}-app"
        - Key: Environment
          Value: !Ref Environment
        - Key: ManagedBy
          Value: CloudFormation
        - Key: Producao
          # !If retorna o primeiro valor se IsProd=true, segundo se false
          Value: !If [IsProd, "sim", "nao"]

Outputs:
  InstanceType:
    Description: Tipo de instancia escolhido pelo Mapping
    Value: !FindInMap [InstanceSize, !Ref Environment, EC2]
  NomeCompleto:
    Description: Nome gerado pelo Sub
    Value: !Sub "${AppName}-${Environment}-app"
  AmbienteProducao:
    Value: !If [IsProd, "SIM - ambiente de producao", "NAO - ambiente nao-prod"]

Passo 2 — Criar e observar os Outputs

  1. No Designer: clique em Create stack → Next
  2. Stack name: techstock-parametrizado → Next
  3. Parameters: Environment → selecione lab · AppName → techstock → Next → Next → Create stack
  4. Aguarde CREATE_COMPLETE → aba Outputs: observe o tipo de instancia e o nome gerado
  5. CloudFormation → Stacks → techstock-parametrizado → Update → Use current template → Next
  6. Mude Environment para prod → Next → Next → Update stack
  7. Na aba Outputs apos o update: AmbienteProducao deve mostrar SIM - ambiente de producao
✅ Esperado: Outputs mudam conforme o Environment: lab usa t2.micro, prod usa t2.medium. O Mapping escolhe automaticamente.
Checklist
Stack techstock-parametrizado criada com Environment=lab
Output InstanceType mostra t2.micro (valor do Mapping para lab)
Apos update para prod: Output AmbienteProducao mostra SIM
Entende a diferenca entre Parameter (entrada do usuario) e Mapping (dicionario interno)

ChangeSet — Veja Antes de Aplicar
ChangeSet ~15 min
ℹ️ ChangeSets sao a feature mais importante para uso seguro do CloudFormation. Eles mostram EXATAMENTE o que vai mudar antes de qualquer recurso ser criado, modificado ou deletado.
💡 Por que ChangeSets existem?
Em producao, executar um update sem saber o impacto e perigoso. Um ChangeSet e o equivalente a um git diff antes do merge — voce ve o que vai mudar e decide se quer prosseguir. Recursos com Replacement=True serao deletados e recriados (downtime!).

Passo 1 — Crie um ChangeSet para a stack existente

  1. CloudFormation → Stacks → techstock-parametrizado → botao Update
  2. Use current template → Next
  3. Mude Environment para dev → Next → Next
  4. Na tela de Review: em vez de Update stack, clique em Create change set for this update
  5. Change set name: update-env-dev → Change set description: Mudando ambiente para dev → Create change set

Passo 2 — Analise o ChangeSet

  1. CloudFormation → Stacks → techstock-parametrizado → Change sets → update-env-dev
  2. Observe a coluna Action: Modify significa modificacao. Add = criar. Remove = deletar.
  3. Observe a coluna Replacement: True = recurso sera DELETADO e RECRIADO (possivel downtime). False = modificacao in-place (sem downtime).
  4. Coluna Logical ID: qual recurso do template sera afetado.
  5. Coluna Physical ID: o ID real do recurso na AWS (ex: i-0abc123).
📸 A tabela do ChangeSet e a informacao mais valiosa do CF. Em prod, nunca execute um update sem revisar o ChangeSet primeiro — especialmente a coluna Replacement.

Passo 3 — Executar ou cancelar

  1. Para aplicar: botao Execute change set → confirme → aguarde UPDATE_COMPLETE
  2. Para cancelar: botao Delete → a stack continua no estado anterior sem nenhuma alteracao
  3. Execute o ChangeSet e aguarde o update completar
  4. Outputs → InstanceType deve mostrar agora t2.small (valor do Mapping para dev)
✅ Esperado: ChangeSet executado com sucesso. Output InstanceType agora mostra t2.small.
Checklist
ChangeSet criado (nao aplicado automaticamente)
Coluna Action e Replacement analisadas antes de executar
ChangeSet executado: UPDATE_COMPLETE
Output InstanceType mudou para t2.small (Mapping de dev)

Drift Detection — Detectando Mudancas Manuais
Drift ~15 min
ℹ️ Drift acontece quando alguem modifica um recurso diretamente no Console AWS sem passar pelo CloudFormation. O CF perde o controle do recurso. Drift Detection identifica essas discrepancias.
💡 O problema do drift
Se voce cria uma EC2 via CloudFormation e depois alguem vai no Console e muda o Instance Type manualmente, o CF nao sabe disso. O codigo diz t2.micro mas a realidade e t2.small. Isso e drift — codigo e realidade divergindo. Em microsservicos e IaC, drift e um problema serio de governanca.

Passo 1 — Introduza um drift manualmente

  1. EC2 → Instances · Localize a instancia techstock-dev-app (criada pela stack)
  2. Selecione a instancia → Actions → Instance settings → Edit tags
  3. Adicione uma tag nova: Key ManualTag Value adicionada-no-console → Save
  4. Agora voce criou um drift: o template CF nao tem essa tag, mas a instancia sim.

Passo 2 — Execute o Drift Detection

  1. CloudFormation → Stacks → techstock-parametrizado → menu Stack actions → Detect stack drift
  2. Aguarde o status mudar de DETECTION_IN_PROGRESS para DRIFTED (pode levar 1-2 min)
  3. Clique em View drift results
  4. O recurso AppInstance aparecera com status MODIFIED (ou DRIFTED)
  5. Clique no recurso → View drift details · Observe: campo esperado vs campo atual para cada propriedade que diverge

Passo 3 — Interpretando os resultados

  1. Property path: caminho da propriedade que divergiu (ex: Tags/2/Value)
  2. Expected value: o que o template CF diz que deveria ser
  3. Actual value: o que realmente existe na AWS
  4. Para resolver: ou atualize o template para incluir a mudanca, ou remova a mudanca manual da AWS
✅ Esperado: Stack mostra status DRIFTED. Drift details exibe a tag ManualTag como propriedade adicionada fora do CF.
⚠️ Em producao, configure um AWS Config Rule para alertar quando drift for detectado. Drift nao resolvido acumula — com o tempo o estado real e o template ficam tao diferentes que o CF nao consegue mais gerenciar a stack.
Checklist
Tag manual adicionada diretamente na EC2 (fora do CF)
Drift Detection executado com sucesso
Stack mostra status DRIFTED
Drift details mostra a tag extra como propriedade nao esperada

Nested Stacks e Outputs Cross-Stack
Nested Stacks ~20 min
ℹ️ Nested Stacks permitem criar modularidade no CloudFormation: uma stack mae que chama stacks filhas. Outputs Cross-Stack permitem que uma stack exporte valores para outra usar — sem hardcode de IDs.
💡 Outputs e !ImportValue
Uma stack pode exportar um valor (Ex: VPC ID) com um nome unico. Outra stack pode importar esse valor com !ImportValue nome-do-export. Isso cria dependencias declaradas entre stacks — e impede deletar uma stack que esta sendo usada por outra.

Passo 1 — Stack de infraestrutura (exporta o SG ID)

  1. CloudFormation → Create stack → With new resources
  2. Template source: Enter template in Designer → Cole o template abaixo
AWSTemplateFormatVersion: '2010-09-09'
Description: TechStock - Stack de Infra (exporta Security Group)

Parameters:
  AppName:
    Type: String
    Default: techstock

Resources:
  SGApp:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupName: !Sub "${AppName}-sg-lab"
      GroupDescription: SG para o app TechStock Lab
      # Usa a VPC default da conta (disponivel no Learner Lab)
      VpcId: !ImportValue techstock-vpc-id
      # Se nao tiver a VPC stack: use a VPC default
      # VpcId: !Ref "AWS::AccountId"  (nao funciona - apenas exemplo)
      SecurityGroupIngress:
        - IpProtocol: tcp
          FromPort: 3000
          ToPort: 3000
          CidrIp: 0.0.0.0/0
          Description: App porta 3000

Outputs:
  SGId:
    Description: ID do Security Group exportado para outras stacks
    Value: !Ref SGApp
    Export:
      # Nome do export deve ser UNICO na regiao/conta
      Name: !Sub "${AppName}-lab-sg-id"

  SGName:
    Description: Nome do Security Group
    Value: !GetAtt SGApp.GroupId
    Export:
      Name: !Sub "${AppName}-lab-sg-name"
ℹ️ Se o !ImportValue techstock-vpc-id falhar (a stack de VPC nao existe no Lab), substitua VpcId pelo ID da VPC default: VPC → Your VPCs → VPC ID da default VPC e use VpcId: vpc-XXXXXXXX como valor fixo para fins de aprendizado.

Passo 2 — Lance a stack de infra-lab

  1. Stack name: techstock-infra-lab → Next → Next → Create stack
  2. Aguarde CREATE_COMPLETE → aba Outputs: copie o valor do SGId

Passo 2 — Stack de aplicacao (importa o SG ID)

  1. CloudFormation → Create stack → With new resources → Designer → cole o template abaixo
AWSTemplateFormatVersion: '2010-09-09'
Description: TechStock - Stack de App (importa SG da stack de infra)

Resources:
  AppInstance:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: ami-0c02fb55956c7d316
      InstanceType: t2.micro
      IamInstanceProfile: LabInstanceProfile
      # !ImportValue busca o Export da outra stack
      # Se techstock-infra-lab for deletada enquanto esta stack existir: ERRO
      SecurityGroupIds:
        - !ImportValue techstock-lab-sg-id
      Tags:
        - Key: Name
          Value: techstock-app-imported-sg

Outputs:
  InstanceWithImportedSG:
    Description: EC2 usando SG importado de outra stack
    Value: !Ref AppInstance
  1. Stack name: techstock-app-lab → Create stack
  2. Aguarde CREATE_COMPLETE
  3. Tente deletar techstock-infra-lab: Stacks → techstock-infra-lab → Delete → observe o erro Export is in use — o CF protege automaticamente!
✅ Esperado: Stack techstock-app-lab usa o SG importado. Tentativa de deletar techstock-infra-lab falha com 'Export is in use' — dependencia declarada e protegida automaticamente.
Checklist
Stack techstock-infra-lab criada com Export do SGId
Stack techstock-app-lab criada usando !ImportValue
Tentativa de deletar infra-lab falhou com Export in use
Entende que !ImportValue cria dependencia protegida entre stacks

Rollback e Troubleshooting
Rollback ~15 min
ℹ️ Nesta tarefa voce vai intencionalmente criar uma stack com erro para ver o rollback automatico e aprender a diagnosticar falhas pelos eventos.
💡 Rollback automatico
Quando qualquer recurso falha durante CREATE, o CloudFormation automaticamente deleta tudo que criou antes de falhar. A stack volta para o estado anterior. Voce NUNCA fica com metade dos recursos criados — ou cria tudo ou reverte tudo.

Passo 1 — Crie uma stack com erro proposital

  1. CloudFormation → Create stack → With new resources → Designer
  2. Cole o template abaixo (tem um erro intencional: SG com descricao com caracteres invalidos):
AWSTemplateFormatVersion: '2010-09-09'
Description: Template com erro intencional para demonstrar rollback

Resources:
  SGComErro:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupName: techstock-sg-erro
      GroupDescription: "SG com caractere inválido — acento e travessão"
      # ERRO INTENCIONAL: a descricao tem acentos e travessao (—)
      # AWS SG so aceita: a-zA-Z0-9. _-:/()#,@[]+=&;{}!$*
      VpcId: vpc-00000000000000000

  EC2Normal:
    Type: AWS::EC2::Instance
    DependsOn: SGComErro
    Properties:
      ImageId: ami-0c02fb55956c7d316
      InstanceType: t2.micro
      IamInstanceProfile: LabInstanceProfile
  1. Stack name: techstock-erro-demo → Next → Next → Create stack
  2. Aguarde — a stack vai falhar em SGComErro

Passo 2 — Diagnostique o erro pelos Eventos

  1. CloudFormation → Stacks → techstock-erro-demo → Events
  2. Filtre por CREATE_FAILED ou role ate encontrar o evento de falha
  3. Observe a coluna Status reason: ela mostra o motivo exato do erro
  4. A stack deve mostrar status ROLLBACK_COMPLETE — significa que falhou E o rollback ocorreu
  5. Note que EC2Normal nao foi criada — o rollback garantiu que nenhum recurso ficou pendente

Passo 3 — Corrija e recrie

  1. CloudFormation → Stacks → techstock-erro-demo → Delete · Uma stack em ROLLBACK_COMPLETE deve ser deletada antes de recriar
  2. Volte ao Designer, corrija a descricao do SG: remova acentos e o travessao (—) → use apenas ASCII basico
  3. Tambem corrija o VpcId: use a VPC ID real da sua conta → VPC → Your VPCs → copie o VPC ID
  4. Recrie a stack com o template corrigido e aguarde CREATE_COMPLETE
✅ Esperado: Stack em ROLLBACK_COMPLETE apos a falha. Apos deletar e recriar com o template corrigido: CREATE_COMPLETE sem erros.
⚠️ Uma stack em ROLLBACK_COMPLETE nao pode ser atualizada — apenas deletada. Isso e diferente de ROLLBACK_FAILED (onde o proprio rollback falhou e requer intervencao manual com continue-update-rollback).
Checklist
Stack techstock-erro-demo falhou com CREATE_FAILED no SGComErro
Events mostra o Status reason com o motivo exato
Stack esta em ROLLBACK_COMPLETE (nao CREATE_FAILED — o rollback ocorreu)
Nenhum recurso ficou criado na AWS (rollback deletou tudo)
Stack deletada e recriada com template corrigido: CREATE_COMPLETE

Limpeza — Deletando Todas as Stacks
Cleanup ~10 min
⚠️ IMPORTANTE: sempre delete as stacks criadas durante atividades de lab. Stacks deletadas removem TODOS os recursos criados — EC2, SGs, etc. Sem custo adicional apos a remocao.

Ordem correta para deletar (dependencias entre stacks)

  1. CloudFormation → Stacks → techstock-app-lab → Delete → Delete stack · Esta usa o SG da infra-lab, entao deve ser deletada PRIMEIRO
  2. Aguarde DELETE_COMPLETE
  3. CloudFormation → Stacks → techstock-infra-lab → Delete → Delete stack
  4. CloudFormation → Stacks → techstock-parametrizado → Delete → Delete stack
  5. CloudFormation → Stacks → techstock-lab-cf → Delete → Delete stack
  6. CloudFormation → Stacks → techstock-erro-demo → Delete → Delete stack (se ainda existir)
  7. EC2 → Instances · Confirme que nenhuma instancia das stacks ainda existe (deve estar Terminated)
✅ Esperado: Todas as stacks com status DELETE_COMPLETE. Nenhuma instancia EC2 ou SG das stacks ainda visivel na conta.
Checklist
techstock-app-lab deletada primeiro (dependencia resolvida)
techstock-infra-lab deletada
techstock-parametrizado deletada
techstock-lab-cf deletada
Nenhum recurso das stacks visivel no EC2 ou VPC
